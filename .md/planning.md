# 📧 AI Phishing Email Detection System - Project Plan

本計畫旨在建立一個能偵測並還原惡意混淆編碼的 AI 釣魚郵件識別系統。透過將 **「解碼還原」** 與 **「AI 紅隊演練」** 融入開發流程，確保系統具備實戰層級的防禦力。

---

## 🏁 第一週：最小可行性架構與基礎解碼 (MVP & Pipeline)
**核心目標：** 打通「輸入 -> 解碼 -> 推論 -> 顯示」的完整資料流，不追求初始準確率。

### 🧑‍💻 組員 A (AI 工程師)
- **Task 1.1 環境建置**：建立 `requirements.txt`。統一環境（Transformers, PyTorch, Streamlit）。
- **Task 1.2 實作推論模組**：撰寫 `inference.py`。載入輕量級模型（如 `distilbert-base-uncased`）作為 Baseline。
- **Task 1.3 API 封裝與校準**：
    - 將模型封裝為 `predict(text)`。
    - **[新增] 實作輸入限制**：限制最大字元長度（如 5000 字元），避免攻擊者利用超長字串進行伺服器壓力攻擊 (DoS)。
- **🚀 本週驗收物**：可獨立執行的 `inference.py`，能針對純文字輸出機率與標籤。

### 🕵️ 組員 B (資安 / 資料工程師)
- **Task 1.4 取得資料集**：下載 Kaggle Phishing Email 資料集（如 Enron），確認格式。
- **Task 1.5 核⼼解碼器實作**：
    - 撰寫 `decoder.py`。實作 **Base64** 與 **URL Encoding** 偵測邏輯。
    - **[新增] 遞迴解碼邏輯**：支援多層編碼（例如 URL 內嵌 Base64），程序將遞迴執行直到字串不再變化。
- **Task 1.6 基礎清洗**：移除 HTML 標籤與不必要的空白字元。
- **🚀 本週驗收物**：具備還原混淆能力、且能處理多層編碼的 `decoder.py`。

### 🥷 組員 C (系統整合與紅隊測試)
- **Task 1.7 前端框架**：使用 Streamlit 建立 `app.py`，包含輸入區、分析按鈕與結果板。
- **Task 1.8 模組整合**：串接 `decoder.py` 與 `inference.py` 至前端。
- **Task 1.9 初始連通性測試**：準備 10 筆內含編碼的手工樣本 `test_payloads.txt`。
- **🚀 本週驗收物**：可在 `localhost` 完整運作的網頁介面。

---

## 📈 第二週：模型微調與特徵工程強化 (Fine-Tuning)
**核心目標：** 訓練專屬模型，提升對「解碼後明文」的判別精準度。

### 🧑‍💻 組員 A (AI 工程師)
- **Task 2.1 資料切分**：撰寫 `train.py`，按 8:2 分割訓練集與驗證集。
- **Task 2.2 模型微調 (Fine-tuning)**：使用「解碼後的明文」重新訓練模型（Fine-tune Transformer）。
- **Task 2.3 多維度效能評估**：
    - 計算 F1-Score, Precision, Recall。
    - **[新增] 誤報率校準**：計算 **False Positive Rate (FPR)**，確保正常郵件不被誤判，並產出 Confusion Matrix。
- **🚀 本週驗收物**：微調後的模型權重檔（於 `models/`）與效能分析報告。

### 🕵️ 組員 B (資安 / 資料工程師)
- **Task 2.4 全量資料處理**：將 Task 1.5 的遞迴解碼器套用到整份資料集。
- **Task 2.5 特徵提取優化**：
    - **[新增] 惡意 Token 化**：偵測到 IP 或特定長網址時，將其替換為專屬 Token（如 `[EVIL_URL]`），輔助模型識別惡意特徵。
- **🚀 本週驗收物**：標準化、高純粹度的 CSV 訓練資料。

### 🥷 組員 C (系統整合與紅隊測試)
- **Task 2.6 推論引擎升級**：將 `app.py` 的後端正式替換為組員 A 訓練出的微調模型。
- **Task 2.7 解碼過程可視化**：在 UI 增設「原始碼 vs 解碼後」對比視窗，增加系統透明度。
- **🚀 本週驗收物**：串接微調模型並具備詳細解碼日誌的 WebApp。

---

## 🛡️ 第三週：AI 紅隊演練與系統驗收 (Red Teaming)
**核心目標：** 以攻擊者視角挑戰系統，找出防禦盲點並完成文件。

### 🧑‍💻 組員 A (AI 工程師)
- **Task 3.1 凍結產線**：停止模型修改，進入測試驗證期。
- **Task 3.2 盲點成因分析**：針對紅隊成功繞過的案例，分析模型 Attention 機制的失效點（如：敏感詞被稀釋）。
- **🚀 本週驗收物**：技術實戰總結報告。

### 🕵️ 組員 B (資安 / 資料工程師)
- **Task 3.3 進階對抗樣本建構**：
    - **[新增] 複雜擾動手法**：實作 **同形異義字攻擊** (Homograph)、**零寬度字元插入**、以及 **大量干擾詞稀釋**。
- **Task 3.4 系統架構圖**：繪製完整的資料流 (Data Flow) 與安全性邊界圖。
- **🚀 本週驗收物**：含括高難度變異載荷的測試集。

### 🥷 組員 C (系統整合與紅隊測試)
- **Task 3.5 執行 AI 紅隊對抗演練**：利用 Task 3.3 的樣本對系統進行壓力測試，尋找系統崩潰界限。
- **Task 3.6 對抗報告紀錄**：紀錄繞過成功的 Payload，撰寫 `Adversarial_Testing_Report.md`。
- **Task 3.7 專案交付**：錄製系統展示影片並整理 README。
- **🚀 本週驗收物**：最終對抗測試報告與 Demo 影片。

## 🛠️ 開發規範與紀律
1. **版本控制**：使用 Git 進行協作。`main` 支線須隨時保持可執行狀態，新功能於 `feature/` 開發。
2. **環境一致性**：統一 Python 版本，相依套件必須鎖定版本號。
3. **[新增] 模型權重管理**：由於權重檔過大，**禁止**直接上傳 Git。請使用 **Git LFS** 列管或存放於共用雲端空間（如 Google Drive/S3），專案內僅保留下載腳本。
4. **防禦優先**：解碼邏輯 (Decoder) 應在 AI 推論 (Inference) 之前，始終假設輸入是具有威脅性的字串。
