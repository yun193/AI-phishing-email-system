# 開發與系統運作行為準則 (System Instructions)

本檔案為開發團隊、系統維護者以及協助開發的 AI 代理在執行任務與編寫程式碼時，必須嚴格遵守的最高指導原則。

## 1. 確保資訊真實與透明 (No Fake Information)
*   **嚴禁捏造資料**：系統在任何情況下（包含例外處理、API 載入失敗等）皆不可偽作成功狀態或填補不實的預設假資料來欺瞞。如果查無資料，必須誠實顯示「無資料」或「載入失敗」。
*   **不可欺騙使用者**：操作成功便返回成功，操作失敗必須明確告知失敗，絕不允許在背景發生錯誤卻在前端依然顯示操作成功的狀況。

## 2. 錯誤處理的明確化 (Explicit Error Handling)
*   **拒絕黑箱，不讓使用者/開發者瞎猜**：系統若發生任何異常，必須拋出或顯示「具體且具可讀性」的錯誤訊息（例如：「找不到此專案 ID」、「結束時間不可早於開始時間」、「權限不足，拒絕存取」）。
*   **呈現詳細的錯誤線索**：發生錯誤時請清楚印出錯誤碼與詳情所在。開發模式中應有完整的錯誤堆疊 (Stack Trace)；正式環境中不論對使用者多友善，其後台日誌 (Logs) 皆必須忠實存放錯誤發生的上下文與細節。
*   **提早發生錯誤 (Fail Fast)**：遇到致命錯誤與非法狀態時應立刻拋出異常並阻斷後續流程，而不是將錯就錯導致更深層的資料污染與不可逆損毀。

## 3. 邊界條件與極端情況的嚴格考量 (Edge Cases & Boundary Defense)
*   **全面性的輸入驗證 (Input Validation)**：針對所有前端表單輸入與後端 API 請求參數，必須實施嚴格的型別檢查與長度、格式限制。防堵可能的 SQL Injection 與 XSS 等攻擊。
*   **空值與時區等極端值處理**：
    *   陣列或字串為空時是否會造成系統閃退崩潰？
    *   跨時區協作時，日期的判定與儲存必須一律遵守 UTC 標準，前端渲染時再依當地時間轉換。
    *   數值超越極限值的時候，系統必須能優雅地阻擋。
*   **多執行緒與並發狀態 (Race Condition)**：若兩位以上的使用者同一秒鐘編輯同一個任務的狀態，系統必須使用樂觀鎖 (Optimistic Locking) 防範，並明確告知後者「此資料已被其他人修改，請重新整理」。

## 4. 行事準則與開發態度
*   **謹慎細心，注重品質**：發布任何變更前，務必再三確認邏輯正確性。測試涵蓋率須達標，任何重構與新增皆應有相應之單元測試保護。
*   **拒絕隨便與無作為**：若發現系統中存在不合理或隱患之邏輯，不能視而不見，請主動提出修改建議並以最嚴謹的方式修復。
